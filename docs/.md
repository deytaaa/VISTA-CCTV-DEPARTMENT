Create a modern web-based Job Order Dispatching and Monitoring System for the City Government of Taguig – CCTV Department using Next.js, Tailwind CSS, Node.js, Express.js, and Supabase PostgreSQL.
The system should follow a real-world government workflow where Job Orders (JO) are created digitally, printed for manual signatures, then uploaded back into the system as proof of completion.

SYSTEM WORKFLOW
STEP 1 — Admin Role (Requesting Office)

Create a Job Order (JO)
Input the following fields:

JO Number (auto-generated, format: JO-2026-0001)
Date
Location
Table 1 — List of Supplies and Equipment: Item No., Item Name, Ref No., Qty (dynamic rows)
Table 2 — Personnel / Job Description: Personnel No., Name (dynamic rows)


Allow dynamic add/remove rows for both tables
Generate a printable PDF version of the JO using @react-pdf/renderer
Send the JO to the Technician role (status changes to Sent)

STEP 2 — Technician Role (CCTV Personnel)

Receive incoming Job Orders via real-time notification (Supabase Realtime)
View JO details and PDF preview
Download and print the JO
Perform the assigned CCTV-related work
Get manual handwritten signatures and fill remaining blank fields on the printed JO paper

STEP 3 — Upload Completion Proof
After completing the task:

Take a picture or scan of the signed JO
Upload the signed JO image or PDF into the system (stored in Supabase Storage)
Add completion remarks
Mark the JO as Completed
Send the completed JO back to the Admin role (status changes to For Approval)
Admin receives real-time notification via Supabase Realtime

STEP 4 — Admin Role Approval

Receive completed JO with real-time notification
Review uploaded proof and signed documents
Approve or Reject the completed work
If Rejected: Add rejection reason/feedback, JO status returns to Processing, Technician receives real-time notification to re-upload
If Approved: JO is archived, final PDF can be printed
Archive approved Job Orders


SYSTEM STATUS FLOW
Draft → Sent → Pending → Received → Processing → Completed → For Approval → Approved / Rejected → Archived

Rejected JOs return to Processing status with rejection remarks visible to the Technician
All status changes are timestamped and logged in the activity log


USER ROLES
RoleAccessadminCreate JOs, approve/reject, archive, view alltechnicianReceive JOs, upload proof, mark completesupervisorView-only access to all JOs, reports, and logs

Implement route guards on both the Next.js frontend (middleware) and Express.js backend (JWT middleware)
UI elements and navigation items must conditionally render based on the logged-in user's role
Supabase Auth handles authentication; use JWT-based sessions with Next.js middleware for protected routes


JO NUMBER AUTO-GENERATION

Format: JO-YYYY-XXXX — example: JO-2026-0001
The numeric sequence resets every calendar year
Generation must be handled server-side via an Express.js endpoint or a Supabase PostgreSQL function/sequence to prevent race conditions when two JOs are created simultaneously
The generated JO number must be unique and collision-free


PDF GENERATION

Use @react-pdf/renderer for all PDF generation
The PDF layout must replicate the official Taguig City CCTV Department Job Order form exactly
The printed PDF pre-fills only the following fields with system data:

JO Number (top right)
Location (top left)
Date (top right)
Table 1: Item No., Item Name, Ref No., Qty
Table 2: Personnel No., Name


All other columns (Description, Remarks, Nature of Work, Signatory line) are printed as blank empty lines — intentionally left for handwriting and manual signatures
This preserves the government manual signature and handwriting workflow
PDF structure must include:

City Government of Taguig header and branding/logo area
CCTV Department label
JO Number, Location, Date (header fields)
Table 1 — List of Supplies and Equipment Needed: columns: No. | Items | Description | Ref No. | Qty | Remarks (Description and Remarks printed blank)
Table 2 — Job Description: columns: No. | Name | Nature of Work | Remarks (Nature of Work and Remarks printed blank)
Signatory line at the bottom (blank for handwritten signature)


Allow both inline preview and download of the PDF
PDF is accessible via a dedicated /jo/[id]/pdf route


FILE STORAGE

All uploaded signed JO proofs (JPG, PNG, PDF) are stored in Supabase Storage
Use a dedicated storage bucket: signed-jo-proofs
The proof_file column in completion_reports stores the Supabase Storage public URL or path — never raw binary data
File upload size limit and accepted MIME types (image/jpeg, image/png, application/pdf) must be validated on both frontend and backend


REAL-TIME NOTIFICATIONS

Use Supabase Realtime subscriptions for live updates across roles
Notification triggers:

Admin sends JO → Technician receives real-time alert
Technician submits completion → Admin receives real-time alert
Admin rejects JO → Technician receives real-time alert
Admin approves JO → Both parties notified


Show in-app notification badges and a notification dropdown in the top header bar
No email notifications required at this stage


SYSTEM FEATURES

Secure login and authentication via Supabase Auth
Role-based access control (admin, technician, supervisor)
Dashboard with statistics and status breakdown cards
Dynamic JO form with two separate add/remove row tables (Supplies and Personnel)

Activity logs and history tracking with timestamps
Real-time updates via Supabase Realtime

Soft-delete on job orders (deleted_at column) — no hard deletes for audit integrity
Responsive professional government-style UI

DATABASE TABLES
users
- id
- name
- email
- password          ← hashed, managed by Supabase Auth, never stored as plaintext
- role              ← enum: admin, technician, supervisor
job_orders
- id
- jo_number         ← format JO-YYYY-XXXX, unique, server-generated
- date
- location
- requestor_name
- status            ← enum: draft, sent, pending, received, processing,
                       completed, for_approval, approved, rejected, archived
- sender_id         ← FK to users
- receiver_id       ← FK to users
- rejection_remarks ← nullable, populated on rejection
- created_at
- updated_at
- deleted_at        ← nullable, soft deletes only
job_order_items (Supplies & Equipment — Table 1)
- id
- job_order_id      ← FK to job_orders
- item_no           ← row number
- item_name
- reference_no
- quantity

Description and Remarks are intentionally omitted — filled by hand on the printed form

job_order_personnel (Job Description / Work Log — Table 2)
- id
- job_order_id      ← FK to job_orders
- personnel_no      ← row number
- name

Nature of Work and Remarks are intentionally omitted — filled by hand on the printed form

completion_reports
- id
- job_order_id      ← FK to job_orders
- remarks
- proof_file        ← Supabase Storage URL/path
- completed_by      ← FK to users
- completed_at

Approval status is tracked exclusively via job_orders.status to avoid sync issues

activity_logs
- id
- user_id           ← FK to users
- action            ← description of the action taken
- job_order_id      ← nullable FK, links log entry to a specific JO
- timestamp
